OVERVIEW · 产品概述
第 2 章　产品概述
2.1 产品定位
以「一个 IP + 一个入口 + 一套 AI 能力」打造苏州轨交「数智文旅第一入口」。苏小T 既是城市数智文旅 IP 形象（讲解员·客服·种草官），又是承载出行、文旅、商业全旅程闭环的 AI 服务体。产品以公众号 H5 为唯一主入口，不另立 App，站台扫码统一回流。

2.3 设计原则
轻量入口：公众号唯一主入口，零安装即开即用。
技术成熟：复用咪咕文旅大模型与智算底座，缩周期、控成本。
硬件解耦：站台能力以软件先行，不依赖硬件采购到位即可上线核心功能。
内容可信：绝不用 AI 编造运营/客服数据，专业内容以客户审定版本为准。
数据闭环：出行+游玩+消费数据打通，沉淀复用、反哺运营。
平滑演进：一二期共用同一底座，二期只做能力叠加与点位扩展。

4.3 智能问答调用链（RAG）
用户自然语言提问（文本/语音，语音先经 ASR 转写）
意图识别：判定出行类 / 文旅类 / 客服类 / 转人工
出行类 → 调用时刻表/到站接口，返回结构化卡片
文旅/客服类 → RAG 检索专属知识库 TopK → DeepSeek 生成
结果渲染为卡片 + TTS 语音播报；低置信或敏感 → 一键转人工
FUNCTIONAL REQUIREMENTS · 功能需求
第 5 章　功能需求（一期详述）
以下为一期功能，采用标准化功能块（编号·优先级·功能描述·业务规则·验收标准·依赖边界），研发可直接拆解、测试可直接编写用例。
5.1 用户体系与实名认证
F-UC-01　统一用户体系与多端身份归一　[P0]
功能描述：基于微信 UnionID 建立统一用户中心（UC），实现公众号/H5/小程序多端身份归一，统一挂接行程、积分、券资源。
业务规则：
游客态：微信静默授权获取 UnionID，免注册即可浏览导览、问答、查询；
会员态：领券/集章/积分等需身份的场景，触发手机号一键实名升级；
一人一账户：同一 UnionID 全端唯一，活动参与一人一号、可核验可溯源。
验收标准：
多端登录后用户身份一致，行程/积分/券数据互通；
游客全流程零注册可用；升级会员无需手动填写信息；
同一用户重复参与活动被正确去重。
依赖/边界：依赖公众号开发者授权。乘车记录关联需票务系统（苏 e 行）开放接口，属可选/分步，不阻塞本功能上线。
F-UC-02　手机号一键实名　[P0]
功能描述：调用微信手机号快速验证组件，用户一键授权即完成实名，符合国家实名制要求。
业务规则：
仅在需要身份的场景触发（渐进式实名），降低流失；
实名信息加密存储，遵循个人信息保护与数据安全合规要求；
实名失败提供重试与人工兜底。
验收标准：
一键授权成功率达预期；实名后会员权限即时生效；
实名信息存储符合合规要求，可通过安全测试。
依赖/边界：需公众号/小程序具备手机号快速验证组件权限。
5.2 智能问答与出行查询
F-QA-01　智能出行查询　[P0]
功能描述：自然语言查询线路、实时到站、末班车、票价、换乘，返回结构化卡片并支持语音播报。
业务规则：
支持文本与语音输入；语音经 ASR 转写后进入意图识别；
出行类意图调用时刻表/到站数据接口；
实时到站若接口未就绪，先按时刻表估算并标注，接口到位后平滑切换；
结果以结构化卡片呈现，关键信息支持 TTS 播报。
验收标准：
常见出行问题（到站/末班/票价/换乘）正确率达标；
响应时延 ≤2 秒；
语音输入可用，播报清晰。
依赖/边界：实时到站需客户提供数据源/接口；时刻表（夏令/冬令）以客户为准。
F-QA-02　7×24 AI 客服　[P0]
功能描述：基于 RAG 专属知识库的全天候智能客服，覆盖票务/失物/设施 + 文旅问答，复杂问题一键转人工。
业务规则：
知识库分三类：运营数据（客户提供）、客服 FAQ（双轨建库）、文旅内容（我方建设、客户审核）；
低置信度或敏感问题自动转人工，并记录会话；
电话/H5/公众号客服口径一致。
验收标准：
问答自主解决率 ≥85%（试运行校准）；
转人工链路通畅，会话上下文可带入；
知识库更新后答案即时生效。
依赖/边界：严禁 AI 编造数据；专业内容以客户审定版本为准。
5.3 AI 行程规划
F-TR-01　个性化单日行程规划　[P1]
功能描述：针对亲子/情侣/银发/青年人群，结合时长与偏好，一句话生成含景点+商户+券+导航的一体化行程。
业务规则：
聚焦一号线、二号线沿线，避免泛数据；
行程项关联可领券与一键导航；
由咪咕文旅大模型 + 线路/景点知识库驱动。
验收标准：
输入人群+时长+偏好可秒级生成合理行程；
行程内景点/商户/券正确关联；一键导航可用。
依赖/边界：多日游与动态调整为二期（P2）。沿线内容以我方建设、客户审核。
5.4 文旅商业融合
F-BIZ-01　文旅商业闭环框架　[P1]
功能描述：上线文旅专线、商户优惠券、景区票务预订、文创展示，打通「出行→游玩→消费」闭环框架。
业务规则：
框架先建、商户分批入驻：谈妥一家上线一家；
线下扫码 → 回流公众号 → 查信息/领券/预约/集章 → 消费沉淀 → 反哺运营；
可量化抓手：专线转化率、券核销率、文创/票务 GMV。
验收标准：
专线/券/票务/文创模块可配置上线；
扫码回流与券核销链路打通；
GMV 与核销数据可在看板追踪。
依赖/边界：商户/景区资源由客户主导洽谈；平台能力一期即建成、零硬件依赖。
5.5 四季 IP 运营
F-IP-01　四季 IP 与互动玩法　[P1]
功能描述：苏小T 春樱-夏荷-秋桂-冬梅四季 IP 矩阵，含 AI 合影旅拍、沿线扫码集章、积分兑换。
业务规则：
AI 合影：选场景上传照片，生成与苏小T/景点的合影海报；
集章：沿线扫码集章，集满指定站点兑限定文创；
积分：文创/票卡/商户券通兑，强化复访。
验收标准：
四季主题可在 B 端切换配置；
合影海报生成成功且风格符合 IP；
集章/积分规则正确，兑换可核销。
依赖/边界：IP 形象授权与活动奖品供给由客户确认；AIGC 生产由我方承担。
5.6 站台轻量化（软件部分）
F-ST-01　站台扫码服务（灯箱二维码）　[P0]
功能描述：狮山核心站台灯箱增设专属服务二维码，实现扫码集章、线上 AI 合影、文旅资讯查询，零硬件采购依赖。
业务规则：
硬件解耦：核心功能仅依赖二维码/立牌，不依赖大屏/一体机；
扫码统一回流公众号 H5，与线上同源数据；
为二期数字人屏端/AR 预留接口。
验收标准：
站台扫码可完成集章/合影/资讯查询；
回流公众号链路通畅；
数据与线上一致。
依赖/边界：需客户提供站台张贴点位与施工许可。站台大屏/一体机为可选升级（客户自采）。
5.7 B 端运营管理平台
F-OP-01　B 端运营配置平台　[P0]
功能描述：面向运营方的自助配置后台：知识库更新、活动配置、券码管理、消息推送，配套可视化数据看板。
业务规则：
内容与资产：知识库/点位/素材/数字资产管理；
运营配置：活动/券/专线/消息推送自助配置，无需技术介入；
权限与数据：角色权限 + 数据看板（咨询量/解决率/转化/核销）。
验收标准：
运营人员可独立完成活动发布、券码管理、消息推送；
数据看板指标准确、可下钻；
角色权限隔离正确。
依赖/边界：需客户指定运营人员与确认指标口径。纯软件、零硬件依赖。

INTERACTION · 交互流程
第 7 章　关键交互流程（供设计参考）
7.1 智能问答主流程
进入公众号 H5 → 文本/语音提问 → 意图识别 → 出行类返回结构化卡片+播报 / 文旅类 RAG 生成 → 满意结束 / 不满意一键转人工。
7.2 渐进式实名流程
游客浏览（免登）→ 触发领券/集章 → 弹出手机号一键实名 → 授权成功升级会员 → 完成领券/集章 → 权益入账。
7.3 行程规划流程
说一句需求 → 选人群/时长/偏好 → 秒级生成方案（景点+商户+券）→ 一键导航出行 / 可调整。
7.4 线上线下闭环流程
线下站台/景点立牌扫码 → 回流公众号 → 查信息/领券/预约/集章 → 消费数据沉淀 → 反哺运营二次触达。
设计交付物：各流程对应页面线框、状态（加载/空/异常/转人工）、组件规范、苏小T IP 视觉规范。

DEPLOY & OPS · 部署运维
第 10 章　部署与运维
10.1 环境规划
开发环境：研发自测，连测试模型 Key 与样例知识库。
测试环境：功能/性能/安全测试，灌入客户审定的样例数据。
生产环境：正式上线，复用咪咕智算底座（GPU/CPU/存储/网络/安全合规）。
10.2 部署要点
微服务容器化部署，API 网关统一入口，配置中心统一管理模型 Key 与端点；
知识库（向量库）与业务库分离部署，素材走对象存储；
站台软件随公众号 H5 上线，无需站台硬件；硬件到位后做屏端软件平滑部署。
10.3 监控与应急
监控：服务可用性、响应时延、问答解决率、转人工率、模型调用量与算力水位。
告警：接口异常、时延超阈、算力超限触发告警。
应急：模型/接口故障降级（备用模型/缓存兜底）；实时到站异常回退时刻表估算。
10.4 运营服务（首年赠送）
含内容持续更新、IP 活动运维、平台系统运维、AI 算力全程支撑；自第二年起年度运营服务费 18 万元。
